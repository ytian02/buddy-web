# 关键设计决策

这份文档用来记录高价值设计决策，避免未来只能靠回忆猜当时为什么这么做。

---

## D-001 第一版不接真实 LLM

- 状态：已采纳
- 原因：先验证 Buddy 作为网页 companion 的产品形态和交互节奏，再决定是否值得接真实模型
- 结果：第一版使用本地规则式回复，但预留 `generateBuddyReply()` 作为统一入口

## D-002 保留 deterministic companion 机制

- 状态：已采纳
- 原因：稳定生成的 bones 有利于长期维护、调试和扩展，也更符合“孵化一个 companion”的产品感觉
- 结果：Buddy 的稀有度、物种和属性通过 seed 稳定生成，soul 存储在本地

## D-003 页面形态采用“宠物 + 聊天区”

- 状态：已采纳
- 原因：这比普通客服页更能体现 companion 感，同时也保留后续接强对话能力的扩展空间
- 结果：当前首页分为 Buddy 展示区与聊天区两部分

## D-004 先用规则回复而不是复杂代理逻辑

- 状态：已采纳
- 原因：当前阶段重点是产品形态和维护节奏，不是复杂 Agent 系统复刻
- 结果：回复逻辑保持轻量，优先保证可读、可替换、可维护
